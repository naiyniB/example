# 当前学习进度

已经学习并理解：

## 1️⃣ 策略模式（Strategy Pattern）

核心理解：

当一个函数有 **多个执行分支** 时：

错误方式：

```js
function pay(type) {
  if (type === "alipay") {
  }
  if (type === "wechat") {
  }
}
```

问题：

- if/else 会越来越多
- 扩展困难

改进：

把每个行为封装成 **策略**

```js
pay(amount, strategy);
```

或

```js
strategy.execute(amount);
```

思想：

**把行为解耦出来，执行函数只负责调用**

---

## 2️⃣ 工厂模式（Factory Pattern）

场景：

当程序需要 **根据条件创建不同对象**

错误方式：

```js
if (type === "alipay") {
  return new Alipay();
}
```

问题：

- 每增加一个类型
- 都要修改创建代码

改进：

使用工厂

```js
factory.create(type);
```

思想：

**把对象创建和业务逻辑解耦**

---

## 3️⃣ 策略 + 工厂组合

理解：

- **工厂负责创建对象**
- **策略负责对象行为**

流程：

```
调用者
   ↓
工厂创建策略对象
   ↓
执行函数调用策略
```

例如：

```
batchPay -> PaymentFactory -> AlipayStrategy
                              WechatStrategy
                              ApplePayStrategy
```

好处：

- 新增支付方式只需要
  - 新策略类
  - 注册到工厂

无需修改核心逻辑。

---

## 4️⃣ 软件工程思想（刚理解）

为什么要拆：

原始代码：

```
执行函数
 + 创建对象
 + 业务逻辑
 + 扩展逻辑
```

问题：

- 高耦合
- 难维护
- 扩展困难

拆分后：

```
执行函数  -> 只负责流程
工厂      -> 负责创建对象
策略      -> 负责具体行为
```

本质：

**职责分离（Separation of Concerns）**

目标：

- 低耦合
- 易扩展
- 易维护

# 当前学习位置

下一步正在学习：

## 观察者模式（Observer Pattern）

学习场景：

**事件通知系统**

例如：

用户注册后：

```
发送 Email
发送 SMS
发送 微信
```

当前问题：

```js
function onUserRegister(user) {
  sendEmail(user);
  sendSMS(user);
  sendWechat(user);
}
```

问题：

- 每新增通知渠道
- 都需要修改代码

目标：

理解如何通过 **观察者模式**：

```
事件发布者
    ↓
订阅管理
    ↓
多个通知订阅者
```

实现：

- 订阅
- 发布
- 自动通知
